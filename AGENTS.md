# 도안호수러닝크루 사이트 작업 지침

## 사용자 기준

- 사용자는 코딩 경력 1개월 정도의 비전공자다.
- 답변은 코드 설명보다 결과 중심으로 한다.
- 어려운 용어는 풀어서 설명하고, 한 번에 너무 많은 선택지를 주지 않는다.
- 디자인 판단은 말로만 설명하지 말고 가능하면 테스트 페이지, 스크린샷, 링크, 시각 비교로 확인시킨다.

## 모델 사용 기준

- 최고 성능 모델을 항상 고정으로 쓰지 않는다.
- 복잡한 디자인 방향 결정, 전체 구조 재설계, 배포 전 최종 판단, 원인 불명 오류 분석에는 고성능 모델을 사용한다.
- 단순 문구 수정, 파일 복사, 링크 교체, 체크리스트 정리, 이미 정한 디자인의 반복 적용에는 가벼운 모델을 우선한다.
- 같은 품질이 유지되면 한 단계 낮은 모델과 짧은 컨텍스트로 작업한다.
- 모델을 바꿔야 할 때는 먼저 이유를 짧게 설명한다.

## 토큰 절약 기준

- 매번 전체 파일을 다시 읽지 말고 `rg`로 필요한 위치를 먼저 찾는다.
- 이미 확정된 내용은 `SITE-CONTENT-CHECKLIST.md`를 기준으로 확인한다.
- 사용자가 방금 확정한 내용만 작게 반영한다.
- 긴 설명 대신 `무엇이 바뀌었는지`, `어디서 볼 수 있는지`, `검증 결과`를 먼저 말한다.
- 디자인 수정은 한 번에 모든 것을 바꾸지 말고, 색상·폰트·레이아웃·콘텐츠 중 하나씩 좁혀 진행한다.

## 품질 유지 기준

- 모바일 카카오톡 브라우저 기준을 최우선으로 본다.
- 공개 사이트에는 게시판, 로그인, 갤러리, 운영진 내부 문서를 넣지 않는다.
- 전화번호는 공개하지 않고 카카오톡 공식채널만 안내한다.
- GitHub push, Railway 배포, 공개 링크 변경처럼 되돌리기 번거로운 작업은 사용자가 명확히 요청했을 때만 진행한다.
- 배포 전에는 `npm run check`와 브라우저 모바일 검증을 실행한다.

## 배포 동기화 규칙 (2026-07-21, 구버전 배포 사고 이후 추가)

Claude Code(여기)에서 작업하든 Codex에서 작업하든, **배포 결과가 항상 GitHub `main`의 최신 커밋과 같아야 한다.**

- 배포는 항상 GitHub `main` push를 기준으로만 한다. `railway up` 같은 로컬 폴더를 그대로 올리는 수동 배포 명령은 사용하지 않는다 — 로컬 사본이 오래되면 이미 배포된 최신 버전을 구버전으로 덮어쓰는 사고가 난다(2026-07-21 실제 발생).
- 배포(또는 배포 확인) 전에는 반드시 아래를 먼저 실행한다:
  ```bash
  git fetch origin main
  git status -sb   # "## main...origin/main" 뒤에 ahead/behind 표시가 없어야 동기화된 상태
  ```
  로컬이 origin/main보다 뒤처져 있으면(behind) 절대 그 상태로 배포하지 않고, 최신 커밋을 반영한 뒤 배포한다.
- 배포 후에는 실제 라이브 주소(GitHub Pages 및/또는 Railway 도메인)에 접속해 방금 반영한 변경 내용이 실제로 보이는지 확인하고 나서 완료로 보고한다. "push했다"만으로 배포 완료라고 말하지 않는다.
- Codex에게 배포를 위임할 때는 이 세 가지(로컬 수동 스냅샷 배포 금지, 사전 동기화 확인, 배포 후 라이브 확인)를 프롬프트에 매번 명시한다.

## 카카오톡 공지 규칙

- 사이트에 실제로 반영되는 업데이트(신규 섹션, 문구·정보 변경, 배포 등)가 있으면, 작업 완료 보고와 함께 **카카오톡 크루방에 올릴 공지 메시지 초안**을 항상 같이 작성한다.
- 별도로 요청하지 않아도 매번 자동으로 작성한다.
- 형식: 짧고 친근한 크루 말투, 이모지 1~2개 정도, `무엇이 새로 생겼는지` + `어디서 보는지(링크 또는 자료실 경로)` + `한 줄 코멘트` 순서.
- 사이트 링크는 확인된 것만 사용한다 — 실제 공개 URL을 모르면 임의로 만들어내지 말고 사용자에게 링크를 물어보거나 자리표시자로 남긴다.
