# 사이트 제작·배포 사전 감사

## 현재 발견된 운영 배포 차단 항목

- 저장소 루트의 기존 사이트에는 예전 예시 일정과 가상 정보가 남아 있음
- R13은 색상 테스트용 단일 페이지이며 전체 사이트가 아님
- R13 사진이 `r9` 폴더에 의존해 폴더를 따로 배포하면 사진이 깨짐
- 메뉴 버튼에 실제 열기·닫기 동작이 없음
- 참여 버튼 일부가 `href="#"`라 이동하지 않음
- `<time datetime="~07:30">` 값이 표준 형식이 아님
- 외부 CDN 폰트에 의존함
- 히어로 PNG가 약 2.2MB라 모바일에서 느릴 수 있음
- `brief.html`은 메뉴에서 숨겨도 주소를 알면 누구나 접근 가능함
- 카카오톡 공유 이미지·사이트 아이콘·사이트맵이 없음

이 항목들은 최종 사이트를 `public/`로 옮길 때 해결해야 하며, 현재 R13을 그대로 Railway에 배포하지 않습니다.

## 이번에 준비된 예방 장치

- Railway에는 `public/`만 포함되므로 과거 시안·회의자료·운영진 문서가 노출되지 않음
- `/healthz`가 정상일 때만 새 배포를 활성화
- HTML은 항상 최신 여부를 확인하고 이미지·폰트는 1시간 뒤 재확인
- gzip·zstd 압축 사용
- 기본 보안·개인정보 보호 응답 헤더 사용
- 없는 주소에는 실제 404 상태와 안내 페이지 제공
- 링크·이미지·중복 id·모바일 설정·테스트 문구·대용량 이미지를 자동 검사
- GitHub Actions에서 공개 파일과 Railway용 Docker 이미지를 매번 검사

## 하네스 기준

### 평상시 제작 중

`npm run check`

- 공개 폴더 필수 파일 확인
- 내부 링크·이미지 경로 확인
- `public/` 밖을 참조하는 경로 차단
- 운영진·비교 페이지 공개 차단
- 모바일 viewport·페이지 제목·중복 id 확인

### 최종 공개 직전

`npm run check:release`

평상시 검사에 더해 다음을 확인합니다.

- 사용자가 최종 배포를 승인했는지
- 테스트·준비 문구가 남지 않았는지
- 카카오톡 공유 정보가 있는지
- favicon과 sitemap.xml이 있는지
- robots.txt의 검색 차단을 해제했는지
- 1.5MB가 넘는 이미지가 없는지

### Railway와 동일한 서버 검사

- Docker 이미지 빌드
- `/healthz` HTTP 200 확인
- 홈 HTTP 200 확인
- 없는 주소 HTTP 404 확인
- 서버 시작 오류 로그 확인

## 아직 진행하지 않은 것

- Railway 프로젝트 생성·연결
- Railway Autodeploy 활성화
- `main` 커밋·push
- 도메인 생성
- 최종 사이트 파일 복사

이 네 단계는 사용자가 명확하게 `배포해줘`라고 말하기 전에는 진행하지 않습니다.
