# R5 Apple-light 설계 및 검증 기준

## 목적

R5의 밝은 정보 안내 성격은 유지한다. 다만 모바일에서 정적인 표처럼 보이지 않도록, 사진·일정·코스가 스크롤에 따라 한 흐름으로 이어지게 만든다.

## 확정된 디자인 규칙

- 중심 축: Hero, 섹션 제목, CTA는 중앙 정렬. 표·일정의 레이블/값은 읽기 위해 좌측 정렬 유지.
- 타이포그래피: SUIT 가변 폰트 하나로 제목·본문의 굵기 차이를 만든다. 별도 장식 폰트는 추가하지 않는다.
- 색: 화이트, 아주 연한 민트, 호수 블루그레이, 소량의 짙은 청록만 사용한다.
- 사진: 실제 도안호수 사진으로 교체 전까지 테스트 풍경은 ‘교체 예정’ 표기를 유지한다.
- 모션: 읽기를 방해하는 자동 이동 금지. 스크롤 진행에 반응하는 1~2개의 시각 신호만 사용한다.

## 구현 구조

### 1. Hero — `index.html .photo-hero`

- 이미지 위에 얇은 빛 레이어(`.hero-light`)를 둔다.
- `data-motion="hero"`를 붙여 스크롤 위치에 따라 이미지가 최대 3% 확대되고 빛이 최대 18px 이동한다.
- 제목과 CTA는 고정된 중앙 축을 유지하며, 텍스트 자체에는 패럴랙스를 적용하지 않는다.
- `prefers-reduced-motion`에서는 정지된 이미지와 빛만 표시한다.

### 2. 다음 정기런 — `.next-run-section`

- 일정 카드에는 `data-motion="schedule"`만 적용한다.
- 화면 진입 시 `opacity`와 `translateY(12px)` 범위에서 한 번만 전환한다.
- 고정(sticky)·자동 스크롤·반복 흔들기는 사용하지 않는다.

### 3. 코스 — `.course-visual`

- 현재 CSS 루프 그래픽을 SVG path로 교체한다.
- `stroke-dashoffset`을 스크롤 진행값에 연결해 0→100% 한 번만 그린다.
- `5KM`은 path가 55% 이상 그려진 뒤 나타난다.
- 모바일에서도 동일하게 동작하되, 실행 범위는 그래픽이 화면 중앙 부근에 있을 때만이다.

### 4. 기술

- `assets/fonts/SUIT-Variable.woff2`: 로컬 폰트, OFL-1.1.
- `assets/lenis.min.js`: 데스크톱 휠 스크롤에만 사용, 모바일 터치에는 사용하지 않는다.
- `script.js`: `IntersectionObserver`로 동작 대상을 등록하고 `requestAnimationFrame`으로 CSS custom property `--progress`만 갱신한다.
- 외부 API, 추적, 로그인, 서버는 추가하지 않는다.

## 모바일 우선 제한

- 680px 이하에서 Hero 이미지 이동량은 1.5% 이하.
- 텍스트는 움직이지 않음.
- 한 프레임에서 transform 하는 요소는 최대 2개.
- 메뉴/FAQ/외부 링크 토스트 기존 동작은 변경하지 않음.

## 검증 루프

### 설계 검증

- [ ] 일정·코스 정보가 애니메이션이 꺼져도 완전하게 읽힌다.
- [ ] ‘예시/확인 전’ 문구가 움직임이나 레이어에 가려지지 않는다.
- [ ] 사진이 실제 도안호수임을 오인하게 하지 않는다.

### 기능 검증

- [ ] 390px, 768px, 1440px에서 가로 넘침이 없다.
- [ ] `prefers-reduced-motion`에서 모든 모션이 멈춘다.
- [ ] 키보드 메뉴·Escape·FAQ·토스트가 기존과 동일하게 동작한다.
- [ ] 모바일 터치 스크롤이 브라우저 기본 물리감을 유지한다.

### 시각 검증

- [ ] 첫 화면에서 Hero 제목·CTA·사진이 모두 보인다.
- [ ] 스크롤 중 텍스트 대비가 WCAG AA 수준을 유지한다.
- [ ] 코스 그래픽은 장식이 아니라 ‘거리/코스’ 정보를 보조한다.
- [ ] 모바일에서 효과가 없는 것처럼 느껴지지 않되, 읽기를 방해하지 않는다.

## 구현 게이트

위 설계를 기준으로 모션 프로토타입을 만들고, 390px/1440px 캡처와 기능 검사에서 모두 통과한 뒤에만 R5에 반영·배포한다.
